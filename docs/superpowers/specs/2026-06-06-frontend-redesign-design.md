# 采集端 App 前端改版设计方案

> 日期：2026-06-06
> 范围：`蓝牙uniapp/`（uni-app + Vue3）采集端 App 的**界面与交互流程**重做
> 状态：设计已与用户逐页确认，待评审 → 进入实现计划

---

## 1. 背景与目标

现有采集端 App 界面粗糙、信息层级混乱，主要问题集中在：登录跳转、连接弹窗、注册页、首页动画、数据可视化页。本次改版目标：

- **更专业**：面向医院医护/技师，达到临床软件应有的清爽、可信观感。
- **理顺使用模型**：把「登录的人=被测的人」的个人 App 模型，改造成**「医护操作员 + 患者档案」的临床多患者模型**。
- **修掉硬伤**：满屏 `0.000000`、一堆 `-` 占位、写死的足底点位文案、暴露在登录页的「网络诊断」按钮等。

### 1.1 受众
医院医护人员 / 技师（操作员）。患者是被测对象，不使用 App。

### 1.2 本次范围 / 不在本次范围

**在范围内（本设计）**：视觉风格系统、组件、各页面布局与交互流程、空态/加载态。

**不在本次范围（后续单独立计划）**：
- **真实数据接入**（图表/实时参数目前用示意数据，接入为改造完成后的计划）。
- **后端模型改造**：操作员/设备档案表、患者表（含自动假名 `subject_id`）、PII 与传感器数据分表、采集数据带「医院/科室/医生+患者编号」、按患者查询历史的只读 API、导出去标识化。本设计依赖这些后端能力，但其实现属于后续计划；前端先按此模型设计，接口未就绪处走空态/示意。

> ⚠️ 后端依赖说明：新模型要求落库时记录采集归属（医院/科室/医生）与患者 `subject_id`，并按患者维度聚合历史。这与 `CLAUDE.md` 阶段 1（全字段落库）、阶段 3（科研只读 API + 去标识化导出）一致，需在数据接入计划中一并实现。

## 2. 视觉风格系统（Design Tokens）

风格方向：**Clinical Light / 医疗清爽**（浅色主题）。

| Token | 值 | 用途 |
|---|---|---|
| `--p` 主色 | `#2F6DF6` | 主操作、强调、折线 |
| `--pl` 主色浅 | `#E7F0FE` | 主色背景/头像底 |
| `--g` 成功/右脚 | `#15A05A` | 正向、右脚序列 |
| `--o` 进行中 | `#E8A33D` | 采集中状态 |
| `--r` 危险 | `#E5484D` | 停止、必填、下降 |
| `--bg` 页底 | `#F5F7FA` | 页面背景 |
| `--bd` 描边 | `#E6EBF2` | 卡片/输入框边 |
| `--t1/--t2/--t3` 文字 | `#0F172A / #64748B / #94A3B8` | 主/次/弱文字 |

- **字体**：`Inter`, `PingFang SC`（无衬线，**不用等宽/Segoe UI**）。
- **圆角**：卡片 12–16px，输入框/按钮 12–13px，手机外框 34px。
- **卡片**：白底 + `--bd` 描边 + 轻投影 `0 1px 3px rgba(16,42,80,.06)`。
- 详见可视化预览 `design-tokens.html`。

## 3. 核心组件

1. **输入框 `inp`**：图标 + 占位/值，统一白底描边圆角；必填项标 `*`。
2. **卡片 `card`**：白底描边圆角，承载各功能块。
3. **指标卡 `m / k`**：标题 + 大数值 + 单位 small + 左/右分列或环比。
4. **状态条 `strip`**：采集中橙色呼吸点 + 计时/条数。
5. **空态 `empty`**：图标 + 标题 + 说明 +（可选）引导按钮，替代所有 `-` / `0.000000` 占位。
6. **当前患者条 `cpbar`**（本次新增）：头像 + 姓名 + 编号 + 「切换」，常驻首页/数据/采集页顶部。

详见 `components.html`。

## 4. 应用模型（核心变更）

```
首次启用（一次性，绑这台设备）:
  医院 + 科室 + 医生姓名 + 联系方式 + 自设口令  → 操作员/设备档案
后续每次打开:
  解锁（输口令）→ 选择 / 新建患者 → 设为「当前患者」
    ├─ 采集     → 数据自动带「医院 / 科室 / 医生 + 患者编号」
    └─ 数据分析 → 看「当前患者」历史
```

- **医护不注册账号、不每次登录**：仅首次「设备启用」自设口令；之后打开只解锁。
- **患者不是账号**：无登录、无密码。仅一条档案（姓名 + 手机号必填 + 性别/年龄选填），系统自动派**假名编号 `subject_id`**（如 `#00427`）。
- **当前患者上下文**：选/建患者后全局锁定，采集与历史均归到该患者。
- **隐私**：姓名仅本机供医护辨认；对外导出默认只带编号（去标识化），PII 与传感器数据分表存储 —— 符合 `CLAUDE.md` 医疗数据红线。
- **安全折中**：设备/科室级一次性口令（方案 B）。既满足「医护不每次登录」，又保证丢设备时数据不裸奔。

## 5. 页面设计

### 5.1 设备启用 + 解锁（替换原「登录/注册」）
- **首次启用**：医院 / 科室 / 医生姓名 / 联系方式 / **自设口令** → 「启用本机」。一次性。
- **解锁页**（之后每次打开）：展示「医院·科室·医生」，仅输口令解锁。提供「重新启用本机」入口。
- 删除原登录页的英文标题、扁平灰输入框、暴露的「网络诊断」按钮。
- 预览：`page-setup-patient.html`（屏 1、屏 2）。

### 5.2 患者选择 / 新建
- **患者选择**：搜索框支持**姓名 / 手机号 / 编号**（手机号用于区分同名患者）；「+ 新建患者」；最近采集列表（头像 + 姓名 + 脱敏手机号 + 编号 + 最近日期 + 采集次数）。点击即设为当前患者。
- **新建患者**：姓名（必填）、**手机号（必填，区分同名）**、性别/年龄（选填）、自动生成编号（只读展示，提示「导出科研只用此编号」）→ 「创建并设为当前患者」。无密码。
- 预览：`page-setup-patient.html`（屏 3、屏 4）。

### 5.3 当前患者条（组件）
- 常驻**首页 / 数据分析页 / 采集（设备）页**顶部：头像 + 姓名 + 编号（性别/年龄）+「切换 ⇄」（回患者列表）。
- 采集页额外显示「本次采集归属：医院 / 科室 / 医生」。
- 预览：`page-setup-patient.html`（屏 5）。

### 5.4 设备连接 & 采集
- **设备模型**：一台设备 = 一双鞋垫（左+右），**只能成对连接**，不支持单只连接。
- 删除不可控的**采样频率**与**单独开关**控制；删除设备名里的「(一双)」。
- **采集中**：状态条只留计时 + 已上传条数 + 两脚信号/条数 + 停止按钮；删除「正在采集（左+右）」文案（采集状态靠橙色 + 顶部「采集中」+ 呼吸点表达）。
- 设备卡字体用 Inter/PingFang（非等宽）。
- 预览：`page-device.html` → `page-device-v4.html`（v4 为最终）。

### 5.5 首页 — 实时步态
- **足底压力分布：沿用原始实现不变**（原 `static/imgs/foot.png` + 原蓝色脉冲点位/CSS ripple 动画）。实现时直接复用现版，不替换为热力图。
- 状态条、实时参数卡（步速/步长/单支撑/双支撑，每项含左/右）按新样式重做。
- 未采集走空态引导，不再露一排 `-`。
- 预览：`page-home.html` / `page-home-v2.html` / `page-home-v3.html`（足底区以原版为准）。

### 5.6 数据分析（按当前患者看历史）
- **顶部时间分段**：本周 / 上周 / 本月 / 近半年，切一次下方所有图表随之变化（不再每个指标各堆三个环形图）。
- **KPI 概览卡（2×2）**：平均步数 / 步速 / 步长 / 双支撑时长，每项带环比（▲▼）。
- **图表**：步速趋势（折线）+ 左右脚压力对比（分组柱）+ 单支撑时间（分组柱）。沿用现有 `qiun-data-charts`(uCharts)，仅规整配色/布局。
- **无数据空态**：彻底替代满屏 `0.000000`。
- 数据按「当前患者」维度调取（依赖后端按患者聚合的历史接口；未就绪前走空态/示意）。
- 预览：`page-data.html`。

### 5.7 我的 / 设置（精简）
- 保留：操作员信息（医院/科室/医生）、设备管理、重新启用本机。
- 移除冗余文案与失效控件。

## 6. 实现方式与顺序

混合式：先建**设计 tokens + 核心组件**，再按优先级逐页改造。

1. 设计 tokens（全局样式变量）+ 6 个核心组件。
2. 设备启用 / 解锁（替换 login/register）。
3. 患者选择 / 新建 + 当前患者上下文（含全局状态、持久化）。
4. 设备连接 & 采集页。
5. 首页实时（足底区复用原实现）。
6. 数据分析页。
7. 我的 / 设置清理。

> 真实数据接入与后端模型改造（操作员/患者/subject_id/归属/按患者历史/去标识化导出）作为后续独立计划，在本前端改版完成后推进。
