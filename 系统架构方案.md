# 智能鞋垫数据采集与云端分析系统 —— 架构评估与改造方案

> 目标：在现有"蓝牙鞋垫 App + 云后端"基础上，构建 **"设备在医院采集 → 数据上传云端 → 研发/科研团队在实验室分析"** 的完整系统。
>
> 本文档先梳理现状架构，再做差距分析，最后给出目标架构与分阶段改造流程。
>
> 代码基线目录：`DataCollectionApp/`（`蓝牙uniapp/` 采集端、`back/` Flask 后端、`admin/` 管理台）。

---

## 1. 结论先行（TL;DR）

**当前项目已经是一套可运行的端到云系统，并非"本地 App"**：

- 采集端（uni-app，Android）已经能通过蓝牙连鞋垫、解析传感器数据，并通过 HTTP 实时上传到云服务器。
- 云后端（Flask + MySQL）已部署在公网服务器，提供注册/登录、设备管理、数据上传、数据查询、SVM 推理等接口。

因此**改造的本质不是"从零搭云"，而是把现有的"消费级健康 App"升级为"可用于临床采集 + 科研分析的数据平台"**。当前最关键的 5 个缺口：

| 缺口 | 严重度 | 说明 |
|---|---|---|
| **传感器数据被大量丢弃** | 🔴 极高 | App 解析出 38 个字段（双脚压力+IMU+步态），但后端只存 6 个（左脚 IMU）。科研团队拿不到压力、右脚、步态数据。 |
| **安全与隐私几乎为零** | 🔴 极高 | 登录**不需要密码**、明文 HTTP、数据库口令硬编码且公网暴露、`SECRET_KEY` 为默认值。涉及患者隐私，不合规。 |
| **缺少科研/实验室读取侧** | 🟠 高 | 没有面向分析的批量导出、队列/时间范围查询、去标识化接口。 |
| **逐条同步上传，不抗高频/弱网** | 🟠 高 | 每个蓝牙通知发一个 HTTP 请求并同步落库+推理；医院网络抖动即丢样本，无本地缓存重传。 |
| **采集与分析耦合** | 🟡 中 | SVM 推理写死在上传链路里，实验室无法独立迭代算法。 |

### 2.1 整体拓扑

```
┌─────────────┐   BLE Notify    ┌──────────────────┐   HTTP/JSON    ┌────────────────────┐

│ BLE 外设     │                  │  Pinia 蓝牙状态机  │                │  JWT鉴权 + SVM推理  │
└─────────────┘                  └──────────────────┘                └─────────┬──────────┘
                                          ▲                                     │
                                          │ 登录/设备/查询                        │ SQLAlchemy
                                  ┌───────┴────────┐                  ┌─────────▼──────────┐
```

> 注意：后端服务（`8.156.78.198`）与数据库（`124.223.217.107`）位于**不同主机**，数据库 3306 端口直接暴露公网（见 `back/config.py:9`）。

### 2.2 组件清单

| 组件 | 技术栈 | 位置 | 职责 |
|---|---|---|---|
| 鞋垫固件 | BLE 外设 | 硬件 | 按设定频率经 NOTIFY 特征值推送 38 个逗号分隔传感器值；接受写入（频率/开关）。 |
| 采集端 App | uni-app + Vue3 + Pinia | `蓝牙uniapp/` | 扫描/连接/订阅蓝牙、解析数据、上传、设备管理、图表展示。 |
| 蓝牙原生模块 | UTS（Android/iOS/Harmony） | `蓝牙uniapp/uni_modules/android-ble/` | 封装 BLE 连接、服务/特征发现、MTU、读写、通知监听。 |
| 后端 API | Flask + Flask-SQLAlchemy + PyJWT + Flask-CORS | `back/` | 用户/设备/数据/周计划接口，JWT 鉴权，SVM 推理。 |
| 机器学习 | scikit-learn SVM（`MultiOutputClassifier`） | `back/api/device.py:611` | 由原始 IMU 推理 T1–T5 概率值。 |
| 数据库 | MySQL 5.7+ | `124.223.217.107` | 5 张表：users / devices / device_raw_data / device_transformed_data / weekly_schedules。 |
| 管理台 | Vue3 + Vite + UnoCSS | `admin/` | 管理员后台。 |

### 2.3 数据流（采集 → 上传 → 存储）

1. **蓝牙接收**（`蓝牙uniapp/store/modules/blueTooth.js:170`）：订阅 NOTIFY，收到 ArrayBuffer → 转字符串 → 按逗号 split，**校验长度必须为 38**。
2. **字段解析**（同文件 `:187-243`）：映射成 38 个字段：
   - 左脚压力 `lp1..lp9`（9 路）
   - 左脚 IMU `ax,ay,az,gx,gy,gz`
   - 左脚步态 `left_step_size / left_speed / left_single_sp_time / left_double_sp_time`
   - 右脚压力 `rp1..rp9`（9 路）
   - 右脚 IMU `right_ax..right_gz`
   - 右脚步态 `right_step_size / right_speed / right_single_sp_time / right_double_sp_time`
3. **上传**（`:246`）：`sendDeviceRawDataApi(device_code, apiData)` → `POST /devices/{code}/raw_data`，**每个通知发一次请求**。
4. **后端落库 + 推理**（`back/api/device.py:188`）：
   - ⚠️ **只取 `ax,ay,az,gx,gy,gz` 6 个字段**（`:198-203`），其余 32 个字段**被直接丢弃**。
   - 同步写入 `device_raw_data`，再调 `DataAnalysisTwo` 跑 SVM 得 T1–T5，写入 `device_transformed_data`。

### 2.4 现有数据模型（`back/models/models.py` / `database_schema_mysql.sql`）

- `device_raw_data`：仅 `ax,ay,az,gx,gy,gz + timestamp`（**单脚 6 维**）。
- `device_transformed_data`：`T1..T5 + timestamp`（SVM 概率输出，语义未在代码中标注）。
- `users`：含 `name/gender/age/address/phone/email/emergency_*`（**患者 PII 明文**）。

## 3. 关键差距分析（为什么现状不能直接当科研系统用）

### 3.1 🔴 数据丢失：采集 38 维，入库仅 6 维
- App 已经辛苦解析了双脚压力、双脚 IMU、步态时空参数（38 维），但 `create_raw_data` 只接收 6 维（`back/api/device.py:198`），DB 也只有 6 列。
- **后果**：实验室永远拿不到压力分布、右脚、步态参数——而这些恰恰是步态/康复分析最核心的数据。**这是改造第一优先级。**

### 3.2 🔴 安全与隐私（医疗数据红线）
- **登录无密码**：`/login` 仅凭"姓名或手机号"即签发 7 天 Token（`back/api/user.py:55-72`）。任何人知道患者姓名即可登录、读取其全部数据。
- **明文传输**：`baseURL = 'http://...'`（`蓝牙uniapp/config/index.js`），`sslVerify:false`（`utils/request.js:49`）。
- **数据库公网裸奔**：3306 直连公网（仍待迁 RDS 内网解决）。口令此前硬编码在 `config.py`，**现已改为读环境变量**（`back/.env`，不入 git）。
- ~~**默认密钥**：`SECRET_KEY = 'your_secret_key'`~~ ✅ **已修复**：`SECRET_KEY` 改为读环境变量、强随机生成，不再用默认值（`back/config.py`）。
- **无审计、无静态加密、PII 明文**：不满足等保/患者隐私要求。

### 3.3 🟠 缺少"实验室分析侧"
- 现有 GET 接口都按"当前登录用户（患者）"维度过滤（`g.user_id`），**没有跨患者队列查询、批量导出、分析友好格式（CSV/Parquet）**。
- 研发团队若想拉全量数据做训练/统计，目前只能直连数据库——既不安全也不可控。

### 3.4 🟠 高频/弱网不可靠
- 逐条 HTTP 上传 + 同步落库 + 同步 SVM 推理（`device.py:188-243`）。多设备高频时是吞吐瓶颈。
- 采集端**无离线缓存/失败重传**：`request.js` 失败只弹 Toast（`:173-221`），样本即丢。医院 Wi-Fi 抖动场景不可接受。

### 3.5 🟡 采集与分析耦合
- SVM 推理写死在上传链路（`device.py:224`），算法每次迭代都要改采集主路径、重启服务，实验室无法独立试验。

## 4. 目标系统架构

### 4.1 设计原则
1. **采集与分析解耦**：上传只负责"可靠落地原始数据"；分析在独立计算侧进行。
2. **原始数据不丢、不改**：原始 38 维全量、不可变（append-only），分析结果单独存。
3. **隐私优先**：传输加密、去标识化、最小权限、可审计。
4. **渐进式改造**：复用现有 Flask/uni-app/MySQL，避免推倒重来。

### 4.2 目标拓扑

```
 医院侧（采集）                          云端（数据平台）                       实验室侧（分析）
┌──────────┐                ┌──────────────────────────────────────┐      ┌─────────────────┐
│ 鞋垫 BLE  │                │  ┌────────────┐    ┌───────────────┐   │      │ 研究员           │
│   │      │   BLE          │  │ 接入网关     │    │ 原始数据存储     │   │      │  Notebook/BI    │
│   ▼      │   ───▶ App ──HTTPS──▶│(鉴权+限流)  │──▶│(raw, 不可变)   │   │◀────│  Python/Pandas  │
│ uni-app  │  批量+离线队列   │  │ 批量摄取API  │    │ MySQL/对象存储  │   │ 只读  │  导出 CSV/Parquet│
│ 本地缓存  │  断点重传        │  └──────┬─────┘    └───────┬───────┘   │ 分析 │                 │
└──────────┘                │         │ 异步消息            │           │ API  └─────────────────┘
                            │   ┌─────▼──────┐      ┌──────▼───────┐  │
   ┌──────────┐            │   │ 分析/推理     │ ───▶ │ 分析结果存储    │  │      ┌─────────────────┐
   │ 医生/管理  │──HTTPS────▶│   │ Worker(SVM等)│      │ transformed   │  │◀────│ admin 管理台      │
   └──────────┘            │   └────────────┘      └──────────────┘  │      └─────────────────┘
                            └──────────────────────────────────────┘
```

### 4.3 数据分层

| 层 | 内容 | 存储 | 可变性 |
|---|---|---|---|
| **Raw（落地层）** | 鞋垫原始 38 维 + 设备/时间/采集会话 | MySQL 宽表 或 对象存储(Parquet) | append-only，不可变 |
| **Curated（治理层）** | 清洗、去标识、重采样、按"采集会话/患者"组织 | MySQL / 数据湖 | 由管线生成 |
| **Analysis（分析层）** | SVM/模型输出、步态指标、统计结果 | `device_transformed_data`（扩展） | 可重算 |

### 4.4 采集端（App）改造要点
1. **全字段上传**：把 38 维全部传给后端（当前已传，只是后端没存）——配合后端/DB 扩列。
2. **批量上传**：本地按"采集会话"聚合，N 条或每 1–2s 打一个 batch，`POST /ingest/batch`，降请求数。
3. **离线队列 + 断点重传**：用本地存储（如 SQLite/`uni.storage`）做环形缓冲；联网恢复后顺序补传；带幂等 ID 去重。
4. **采集会话（session）**：每次"开始采集"生成 `session_id`（患者×设备×时间段），便于实验室按会话取数。
5. **时间基准**：上报"设备/采集时刻"时间戳，而非到达服务器时间（现状用 `datetime.now()` 服务器时间，`device.py:217`，高频下不准）。

> **进展（2026-06-07）**：采集端 UI/交互模型改版已实现（分支 `feature/frontend-redesign`，详见 CLAUDE.md §2.1）。已把「个人 App 账号登录」改为「设备级操作员启用/解锁 → 选/建患者 → 当前患者上下文」，并统一视觉令牌、去除 `0.000000`/`-` 占位（改空态）。本线仅做**视觉 + 交互流程**（示意数据/空态）；上述 1–5 项（全字段上传、批量、离线队列、session、时间基准）与后端「操作员/患者表、`subject_id`、PII 分表、采集归属」仍属**后续计划**，与阶段 1/3 合并落地。

### 4.5 云端摄取服务改造要点
1. **新增批量摄取接口** `POST /ingest/batch`：接收一个会话的多条样本，批量 `bulk_insert`，**只负责落库，不在请求内做推理**。
2. **推理异步化**：落库后投递消息（轻量可用 DB 轮询表/RQ/Celery + Redis），由独立 Worker 跑 SVM 写分析层。采集链路与算法解耦。
3. **存储选型**：
   - 起步：MySQL `device_raw_data` 扩成 38 列宽表（改动最小，与现有栈兼容）。
   - 规模化：原始时序写对象存储（MinIO/OSS）成 Parquet，MySQL 只存索引/会话元数据。
4. **幂等**：摄取按 `(session_id, seq)` 唯一约束，重传不产生重复。

### 4.6 实验室分析侧（新增，关键）
1. **科研只读 API**（独立角色，最小权限）：
   - `GET /research/sessions?patient=&device=&from=&to=` 列采集会话。
   - `GET /research/export?session_id=&format=parquet|csv` 批量导出（异步生成下载链接）。
   - `GET /research/raw?from=&to=&fields=` 跨患者按时间/字段取数。
2. **去标识化**：导出默认用 `subject_id`（假名）替代姓名/手机号；真实身份映射单独存、单独授权。
3. **分析环境**：实验室用 Python/Pandas/Jupyter 直接读导出的 Parquet，或连只读副本库；模型在分析层迭代，成熟后再回灌为 Worker。
4. **不让实验室直连生产库**：通过只读 API / 只读副本，避免误操作与隐私扩散。

### 4.7 安全与合规（必须项）
| 项 | 措施 |
|---|---|
| 传输 | 全链路 HTTPS（云端配 TLS，App 关闭 `sslVerify:false`）。 |
| 认证 | 登录加**密码/短信验证码**；`SECRET_KEY` 改强随机并入环境变量；Token 缩短有效期 + 刷新机制。 |
| 授权 | 角色分离：患者 / 医生 / 管理员 / **科研只读**；科研角色禁访 PII。 |
| 数据库 | 3306 不暴露公网（安全组仅放行后端内网）；口令进环境变量/密钥管理；启用最小权限账号。 |
| 隐私 | PII 与传感器数据分库/分表；导出去标识；建立访问审计日志。 |
| 合规 | 按医疗数据/个人信息保护要求做留存、脱敏、知情同意与权限审批流程。 |

### 10.2 可以自己再租一台服务器迁过去吗？

**完全可以，且强烈建议——这就是当前正在执行的方案（阶段 0.5）。** 服务器地址不在深层耦合处，迁移成本很低：

| 步骤 | 改动 |
|---|---|
| 1. 开通服务器 | 阿里云 **ECS + RDS（MySQL）**，放**同一 VPC、同可用区**；RDS 走内网、不开公网 3306（比自建 MySQL 更稳更安全）。2核4G 起步够用。 |
| 2. 建库 | RDS 创建数据库 `lanya`，导入 `back/database_schema_mysql.sql` 建表。**当前尚未采集数据，无历史数据需迁移**，无需 `mysqldump`。 |
| 3. 改后端配置 | 复制 `back/.env.example` 为 `back/.env`，填入新 RDS 内网地址/新口令/强随机 `SECRET_KEY`（连接串已由 `config.py` 从环境变量拼装，无需改代码）。 |
| 4. 部署后端 | 将 `back/` 部署到新服务器，`pip install -r requirements.txt` 后启动；生产环境用 gunicorn + nginx，**不要用 `app.run`**。 |
| 5. 改 App 指向 | 修改 `蓝牙uniapp/config/index.js` 的 `baseURL` 为新服务器地址，**重新打包 Android App**。 |

**关键提醒**：
- 第 5 步意味着**手机上的 App 必须重新编译安装**——服务器地址是打包进 App 的；旧 apk 仍会往旧服务器发数据。
- 迁移时建议顺手做两项安全改造（即"阶段 1"内容）：① 用域名 + HTTPS 替代裸 IP + http；② 数据库只监听内网/本机，关闭公网 3306。
- **服务器地域与备案（已定）**：**留大陆节点 + 备案，不用海外节点**。采集在上海、数据属医疗个人敏感信息，落境外触发**数据出境安全评估/单独同意（PIPL）**+ 伦理/医院合规风险 → 排除海外。**免费试用 ECS 不满足备案**（须**大陆 + 包年包月 ≥3 月**），需把生产 ECS 换/续成合规实例（非另购备用机）。链路：医疗采集须 HTTPS → 须域名 → 大陆机域名须备案；测试期用 `http://IP` 不需域名/备案，可与备案（约 1–2 周）并行。详见 [迁移操作清单.md](迁移操作清单.md) 第 5 步。
